    异常类型
    说明
    脚本异常

    EVO在数据中心内的用户群体划分

     

    在数据中心内部，EVO的用户群体可划分三个：操作用户、定义用户和审批用户。

    用户
    职能

    说明

    操作用户
    通过EVO平台来直接申请、分配、使用和管理资源和服务。
    是管理决策的最终执行者和技术工具的最终使用者。在“行动上”决定了资源和服务会被怎样使用。

    操作用户一般为数据中心的一线运维人员，通过EVO平台提供的各类软件工具（比如巡检）来开展其运维活动，达成运维目标。

    因此从操作用户的视角来看，如果其运维操作的主要目的是分配资源，那么EVO是IaaS平台，如果其运维目标是提升已分配资源的效益，那么EVO是一个SaaS平台。

     

    定义用户
    通过技术定义资源和服务在平台中应该如何被使用。包括：
    （1）平台开发者；
    （2）平台应用开发者。
    定义用户一般为数据中心的运维开发人员，通过EVO平台来进行各类资源、服务的分配或实现。
    特别地，从平台应用开发者的视角看，EVO是PaaS平台。
    审批用户
    从管理维度定义资源和服务在平台中应该如何被使用。
    负责对最终用户、定义用户的操作进行审批、复核。
    审批用户是各类资源、服务功能实现、分配和使用过程中的秩序维持者。
    一般由企业组织的领导或技术条线专家组成。

     

    对于大中型数据中心，EVO平台的用户由以上三大群体构成。

    对与小型数据中心，根据企业文化、技术能力和人员配备情况等的不同则会有不同的组合和侧重，部分组织还会选择全部或部分外包定义用户类的职能。

    随着DevOps文化的普及，部分组织甚至会将审批权限下放，采用“互审批”模式，在确保合规的前提下提升敏捷性。

     

     

     

     

    EVO（操作系统）作业服务支持的语言类型

     

    EVO通过模拟真实用户的CRT、CMD终端登陆行为来实现对对各类语言环境的支持。

     

    以一个以root用户纳管的Linux操作系统为例，Agent（广义上还包括Agentless）的行为和用户以root账户通过CRT终端登陆是基本相同的，EVO会加载相应的环境变量、获得同等的管理权限。这样做的好处是用户不必进行特殊的转换即可将管理操作转移到EVO平台之中。

     

    和任何自动化框架一样，如果需要将交互式脚本的功能移植到自动化平台中，须首先进行非交互化改造。

     

    以下列出了在运维领域典型的脚本语言类型，用户可根据操作系统环境的差异选择适用的语言进行运维自动化开发：

    文件后缀
    语言类型

    说明

    .sh
    Shell
    		 EVO在Unix环境下默认调用的类型。根据操作系统的设置差异，最终使用的可能是Bash、Ksh等脚本类型。如果需要使用明确的脚本解释器，需要在Shebang中指定。比如#!/bin/bash。
    .bash
    Bash
    Linux 默认
    .ksh
    Ksh
    AIX 默认
    .bat

    .pl

    一般情况下，Linux、AIX、HP-UX均默认具有Perl的运行环境。
    .py
    Python
    Linux一般具有Python运行环境。
    .rb
    Ruby
     
    .vbs
    VBScript

    .ps1

    一般具有PowerShell运行环境，早期的Windows操作系统，如 Windows 2003需要自行安装。

    通过观察上表可以发现，EVO的作业服务对各类编程语言的支持实际上取决于操作系统所具备的语言环境。如果用户需要管理Linux、AIX、HP-UX和Windows等多种平台，那么，就需要学习各类编程语言的特性，这提高的运维开发的复杂性和难度。

     

    为解决这一问题，Puppet和Ansible等均构建了一套自成体系的编程语言来屏蔽系统间的差异性。但这又将自动化运维开发带入专业化的方向，需要专门的人员才能取得较好的效果。

     

    解决这一问题的建议是为各个系统平台安装统一的编程语言环境，比如Python。

    附 流程编辑器说明与分支判断实例

     

    1. 开始事件

    开始事件用于流程（或子流程）的开始，形状上为一个细线圆圈，圈内图例表明开始事件的具体类型。

    图标
    使用说明
    空的流程启动事件
    使用一个定义好的消息名称来启动流程实例（收到其他流程发送的消息），消息开始事件只支持顶级流程，消息开始事件不支持内嵌子流程，支持调用子流程。
    流程部署之后会定时触发流程的启动执行，不能用于子流程
    可以通过一个定义好的信号名称来触发当前流程实例的启动
    只能用来触发一个事件子流程，不能用来启动流程

     

     

    2. 活动

    图标
    使用说明
    需要人工审批的任务节点，需要选择审批人和审批模板
    应用发布独有，下发文件到服务器列表的指定位置
    应用发布独有，在指定的服务器执行脚本，在执行前可以设置参数

     

     

    3. 结构模块

    图标
    使用说明
    子流程（Sub-process）是一个包含其他节点，网关，事件等等的节点。 它自己就是一个流程，同时是更大流程的一部分。 子流程是完全定义在父流程里的 （这就是为什么叫做内嵌子流程）。
    .子流程运行过程中抛出的事件，可以被子流程边缘定义的 边界事件捕获， 这样就可以创建一个仅限于这个子流程的事件作用范围。
    使用子流程要考虑如下限制：
    子流程只能包含一个空开始事件， 不能使用其他类型的开始事件。子路程必须 至少有一个结束节点。
    调用子流程。
    调用当前流程定义外部的一个流程，把外部流程当做一个分支执行，默认创建并行子流程，当前流程等待子流程执行完成再继续执行

     

    4. 网关

    图标
    使用说明

      
    排他网关。
    当流程执行到这个网关，所有外出顺序流都会被处理一遍。 其中条件解析为true的顺序流（或者没有设置条件，概念上在顺序流上定义了一个'true'） 会被选中，让流程继续运行。
    将流程分成多条分支，也可以在分支执行完成之后把多条分支汇聚到一起。

    5. 边界事件

    图标
    使用说明
    执行到某个节点时设置定时启动分支流程。当流程执行到绑定了边界事件的环节， 会启动一个定时器。 当定时器触发时（比如，一定时间之后），环节就会中断（可以设置为中断或不中断）， 如果设置为中断则中断当前流程，沿着定时边界事件的外出连线继续执行。如果设置为不中断，可以当前流程和定时事件的外出执行流程可以同时继续执行。
    节点边界上的中间捕获错误事件， 它会捕获当前节点范围内抛出的错误。大多用于内嵌子流程， 或调用节点，对于子流程的情况，它会为所有内部的节点创建一个作用范围。 错误是由错误结束事件抛出的。 这个错误会传递给上层作用域，直到找到一个错误事件定义向匹配的边界错误事件。当捕获了错误事件时，边界任务绑定的节点就会销毁， 也会销毁内部所有的执行分支 （比如，同步节点，内嵌子流程，等等）。 流程执行会继续沿着边界事件的外出连线继续执行。

     

    6. 结束事件

    结束事件用于流程（或子流程）的结束，形状上为一个粗线圆圈，圈内图例表明结束事件的具体类型。

    图标
    使用说明

      
    结束当前流程或分支并抛出取消事件。
    取消结束事件只能与BPMN事务子流程结合使用。 当到达取消结束事件时，会抛出取消事件，它必须被取消边界事件捕获。 取消边界事件会取消事务，并触发补偿机制。我们现在没有用到事务子流程（被隐藏）。
    结束当前流程或分支并抛出错误。
    流程的当前分支就会结束，并抛出一个错误，这个错误可以被对应的中间边界错误事件捕获。 如果找不到匹配的边界错误事件，就会抛出一个异常。
    空结束事件。
    意味着到达事件时不会指定抛出的结果，引擎会直接结束当前执行的分支，不会做其他事情。
    终止当前流程。
    当遇到某种原因流程无法继续执行时，可以直接指向当前节点，直接终止当前的流程实例
